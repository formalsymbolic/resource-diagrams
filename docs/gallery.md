# Diagram Gallery

> **Note**: Diagrams and text here are illustrative examples from a symbolic model. They are not security analyses or formal proofs. See the [README](../README.md) for scope and limitations.

This gallery curates the committed diagram exports (01–09 in `examples/exports/`) that illustrate how Resource Diagrams make **policy and tool-definition copying (Δ forks on program triangles ▼) versus one-way channels** visible as explicit geometry in the string-diagram notation.

The central modeling device is the Δ/⊤ distinction: Δ (fork nodes) for duplication of copyable program-like elements (policies, tool definitions) and ⊤ (stem nodes) for linear/one-way use of transient resources (user inputs, observations). This makes distinctions that are often implicit in code into explicit, countable structures in diagrams.

## Interpreting the Diagrams (How to Use These)

These .mmd files are self-contained Mermaid sources (GitHub renders them directly; also viewable in the Mermaid Live Editor or VS Code). They are generated by the public API (see runnable examples below) and serve as visual references for the structural patterns.

**Visual conventions in these diagrams**:
- **▼ triangles** (often orange/pink in styling): program-like elements (policies, tool defs, sub-policies) treated as basic data on the Ξ object; these support explicit Δ copying.
- **Black-dot forks (Δ)**: explicit duplication points (e.g., policy copied into multiple steps or agents).
- **Gray stems (⊤)**: termination / delete after linear use; absence of Δ on a wire indicates one-way flow.
- **Boxes**: computation steps or agent actions (may carry cost annotations on parallel resource wires).
- **Wires/arrows**: flow of data or resources; parallel wires can represent token budgets or other accounting.

**Interpreting Δ/⊤ geometry for modeling use cases**:
- Policy forks with few stems on sensitive paths can make potential persistence visible in the model.
- Explicit stems on user data after use show termination of one-way channels.
- Separate policy triangles per component make independence explicit; shared state appears via controlled Δ.
- `analyze_safety_geometry` counts forks and stems by type. This is a structural aid for reviewing diagram geometry.

**How to use these**:
- All linked examples are runnable from a fresh clone with `python examples/<name>.py` (bootstrap in each script adds src/ to path; no install required).
- Cross-link the exports back to their generators: `examples/diagram_export.py` for the 01–04 paper reconstructions; the specific `examples/*_agents.py`, `react_loop.py`, `reflexion_critique.py`, `multi_agent_coordination.py`, `guarded_contrast.py` etc. for the agent patterns (05–09 and contrasts).
- Use the diagrams as discussion aids when reviewing agent scaffold designs, transcribing graphs (see `docs/recipe_langgraph.md`), or exploring the monoidal constructions from the source papers.
- To regenerate or extend: the models builders in `src/resource_diagrams/models/agents.py` (and Diagram.to_mermaid) are the source of truth for 05+.

## Foundational Paper Reconstructions (01–04)

These four are generated live by the diagrams layer alone (no models layer) via `examples/diagram_export.py`. They reconstruct key structures from the Monoidal Computer papers as reference; they demonstrate the primitives (composition, evaluator law, fixed points via copying, and the comonoid Δ/⊤) that the agent models build upon.

### 01: Basic String Diagrams
- **Export**: `examples/exports/01_basic_string_diagrams.mmd`
- **Generator**: `examples/diagram_export.py` (MermaidRenderer.render_basic_monoidal)
- **Why the Δ/⊤ geometry matters for modeling**: This diagram introduces the core visual vocabulary—sequential and parallel (tensor) composition, unit laws, and the triangle notation for elements as morphisms from the monoidal unit. The later power of the approach rests on treating certain objects (programs) as "basic data" that inhabit the subcategory supporting Δ and ⊤; without this foundation the explicit forks and stems used in agent models would have no formal grounding in the symbolic model.

### 02: Universal Evaluator Law
- **Export**: `examples/exports/02_universal_evaluator_law.mmd`
- **Generator**: `examples/diagram_export.py` (MermaidRenderer.render_evaluator_law)
- **Why the Δ/⊤ geometry matters for modeling**: The evaluator u turns a program-as-data triangle (copied or referenced via the data-service structure) into the corresponding computation box. The diagram makes the "p as data on the left, {p} as behavior on the right" relationship graphical; this is the mechanism that lets the fixed-point construction (03) and agent policy self-reference work by feeding copied program triangles into the evaluator.

### 03: Fixed Point Construction
- **Export**: `examples/exports/03_fixed_point_construction.mmd`
- **Generator**: `examples/diagram_export.py` (MermaidRenderer.render_fixed_point_construction) and also `examples/fixed_point_demo.py`
- **Why the Δ/⊤ geometry matters for modeling**: Because programs are basic data (supporting the Δ axiom δ ∘ p = p ⊗ p), a program triangle can be forked once and one copy fed back into the evaluator that interprets it. The short diagrammatic argument for "every computation has a fixed point" therefore becomes visually immediate. In agent modeling this directly supplies a handle on self-referential or reflective policies (see also the agent-policy fixed-point variant 06).

### 04: Data Services Comonoid
- **Export**: `examples/exports/04_data_services_comonoid.mmd`
- **Generator**: `examples/diagram_export.py` (MermaidRenderer.render_data_service_comonoid)
- **Why the Δ/⊤ geometry matters for modeling**: This is the "classical data" structure (commutative comonoid) that distinguishes the model: objects in the data-service subcategory preserve Δ (copy) and ⊤ (delete). Programs live here, which is why policy triangles can be duplicated freely while user observations typically do not; the comonoid laws are what make the Δ forks in all the agent diagrams (05–09) structurally valid rather than ad-hoc.

## Agent Modeling Patterns (05–09 + supporting examples)

These use the higher-level builders in `resource_diagrams.models` (which in turn use the diagrams + core layers) and are committed snapshots of what the builders produce. They focus on making the copy-vs-linear distinction legible for common agent scaffolds.

### ReAct / Tool-Use Loop (Primary Demonstration)
- **File**: `examples/react_loop.py` (runnable with `python examples/react_loop.py`)
- **Export**: `examples/exports/05_react_agent_scaffold.mmd`
- **Generator**: `examples/react_loop.py` (and `build_simple_react_diagram` in models)
- **Why the Δ/⊤ geometry matters for modeling**: Policy and tool definitions (▼ triangles) are explicitly forked (Δ) into every reasoning step and tool invocation, while the user query and subsequent observations travel on one-way wires (no Δ). The geometry therefore makes the persistence of policy fragments across cycles visible as a structural fact of the diagram; memory states receive Δ copies of policy but not of the original query. This is the core illustrative pattern the library uses to explore information-flow distinctions in ReAct-style scaffolds as a review aid. The diagram also carries real `StringDiagram` objects and `analyze_safety_geometry` counts for the forks vs. stems. (Older "Safety Text" annotations in the export are illustrative only.)

### Agent Policy Fixed Point (06)
- **Export**: `examples/exports/06_agent_policy_fixed_point.mmd`
- **Generator**: snapshot from models-layer usage of the fixed-point renderer (see `diagram_export.py` comments and `fixed_point_demo.py`)
- **Why the Δ/⊤ geometry matters for modeling**: The same Φ construction from 03 is applied to an "agent_policy" triangle. The Δ fork lets the policy copy itself and be fed to the evaluator, yielding a diagrammatic fixed point. This supplies a clean visual model for self-referential or potentially self-modifying agent policies within the symbolic framework; resource wires (e.g. token budgets) can be shown in parallel to illustrate consumption during self-application.

### Explicit Reflexion with Separate Critic Policy Fork + Resource Accounting (NEW structural pattern)
- **File**: `examples/reflexion_critique.py` (runnable)
- **Export**: `examples/exports/08_reflexion_critic_policy.mmd`
- **Generator**: `examples/reflexion_critique.py` (via `build_reflexion_with_critic_diagram`)
- **Why the Δ/⊤ geometry matters for modeling**: The actor maintains its own policy Δ while observations cross to the critic on a dedicated one-way wire (no Δ at the boundary). The critic has an *independent* CriticPolicy triangle with its own Δ fork, making the distinct "critic policy" an explicit first-class resource rather than an implicit string. Stems terminate both the critic's input observation and its revision output; the critique step itself carries an explicit cost annotation. The analyzer therefore reports two families of policy forks plus the extra resource surface ("critic tax"). The geometry provides a structural way, inside the symbolic model, to express and count the information boundary and duplicated policy cost that appear in self-critique / Reflexion-style loops.

### Guarded vs Unguarded One-Way Channels (Core Hook, Quantified)
- **File**: `examples/guarded_contrast.py` (runnable with `python examples/guarded_contrast.py`)
- **Generator**: `examples/guarded_contrast.py` (via `build_guarded_vs_unguarded_contrast` in models)
- **Why the Δ/⊤ geometry matters for modeling**: Both diagrams have identical policy/tool Δ forks (the agent can still use its tools across steps). The sole structural difference is the presence of an explicit Stem (⊤) on the sensitive one-way wire after first use in the guarded case. The analyzer directly surfaces the delta via `stems` count and `has_explicit_guards` flag. In the modeling use case this makes "guarding" a diagrammatic act—terminating the channel for transient user data—legible and countable without changing the copyable policy surface. It is a distilled illustration of how the geometry can express use-once policies for credentials, PII, or observations in the symbolic model.

### LangGraph-Style Graph Transcription
- **File**: `examples/langgraph_style_transcription.py` + `docs/recipe_langgraph.md`
- **Generator**: `examples/langgraph_style_transcription.py`
- **Why the Δ/⊤ geometry matters for modeling**: The transcription maps a typical `StateGraph` entry binding (persistent tools/prompt) to an initial Δ on the policy triangle at "compile" time; subsequent nodes and conditional observation edges become boxes and one-way wires. The resulting diagram + analyzer yield the same policy-fork / one-way metrics as the native builders. This extends the review-aids value to the large LangGraph user community: the diagrams can be produced and inspected with zero LangGraph runtime dependency, using only the structural model.

### Information Flow Contrast
- **File**: `examples/info_flow_diagram.py`
- **Generator**: `examples/info_flow_diagram.py`
- **Why the Δ/⊤ geometry matters for modeling**: A direct side-by-side (or overlaid) contrast that renders copyable internal state via Δ forks versus user secrets/observations via linear flow and ⊤. Useful as a minimal teaching example of the core distinction the whole library is built around.

### Hierarchical / Nested Agents (NEW)
- **File**: `examples/hierarchical_agents.py` (runnable)
- **Export**: `examples/exports/07_hierarchical_nested_agents.mmd`
- **Generator**: `examples/hierarchical_agents.py` (via `build_hierarchical_agent_diagram`)
- **Why the Δ/⊤ geometry matters for modeling**: Inner agents appear as first-class resources: explicit sub-policy triangles (▼ SubPolicy:*) tensor-ed at the supervisor entry. The supervisor can Δ-fork a sub-policy (copy the sub-agent definition for delegation/reuse) or pass it linearly; after the inner run, stems are placed on both the sub-observation *and* on the sub-policy resource itself. The tree structure plus per-sub copy-vs-linear + post-use stems make nested lifetimes visible in one diagram. The analyzer classifies multiple policy forks (supervisor + subs under the policy type) and stems_by_type including policy stems. For the modeling use case this gives a structural vocabulary for "agent spawning" or "sub-agent code as data" patterns, letting a reviewer see exactly which sub-policies get copied versus consumed and terminated.

### Multi-Agent Coordination with Boundaries (NEW)
- **File**: `examples/multi_agent_coordination.py` (runnable)
- **Export**: `examples/exports/09_multi_agent_coordination.mmd`
- **Generator**: `examples/multi_agent_coordination.py` (via `build_multi_agent_coordination_diagram`)
- **Why the Δ/⊤ geometry matters for modeling**: Each agent has a private policy triangle, so Δ forks are independent (one agent's policy copy does not duplicate another's). Private thoughts are stemmed after local work (isolation enforced in the model). The shared blackboard receives Δ only at explicit access points (copyable common state). The only crossing point is the explicit ConsensusMerge box. Final stem on the shared resource. Analyzer reports forks_by_type (policy for the agents, data/context for shared) and stems on sensitive (privates) vs. data; flow tracking can show whether any private path reaches the merge. The geometry therefore models blackboard, debate, or MoA architectures with auditable boundaries inside the symbolic model: private surfaces are terminated by stems, shared surfaces are controlled by the merge rule and final stem.

### Resource Accounting
- **File**: `examples/resource_trace.py`
- **Generator**: `examples/resource_trace.py`
- **Why the Δ/⊤ geometry matters for modeling**: Token budgets and costs appear as first-class parallel resource wires that interact with the copied policy paths (Δ on policy can be accompanied by cost accumulation on the resource wire). This extends the core geometry to make consumption alongside control flow visible.

### Fixed-Point Construction (Paper I Foundation) — agent context
- **File**: `examples/fixed_point_demo.py`
- **Export**: `examples/exports/03_fixed_point_construction.mmd`
- **Generator**: `examples/fixed_point_demo.py` (and diagram_export)

## How to Build Your Own
All diagrams above are generated from the public API. Run the corresponding example scripts from a fresh clone — no installation required thanks to bootstrap guards.

Example usage of the builders:
```python
from resource_diagrams.models import (
    build_simple_react_diagram,
    build_hierarchical_agent_diagram,
    build_reflexion_with_critic_diagram,
    build_multi_agent_coordination_diagram,
)
from resource_diagrams.diagrams.safety import analyze_safety_geometry

d = build_simple_react_diagram(tools=["search", "code"], cycles=2)
print(d.to_mermaid())
print(d.safety_explanation)

# New patterns
dh = build_hierarchical_agent_diagram(["sub1", "sub2"])
print(analyze_safety_geometry(dh.string_diagram))  # policy_forks for sup+subs, stems on subpolicies
```
